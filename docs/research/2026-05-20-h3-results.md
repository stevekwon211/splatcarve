# H3 — Stack results (Wave D.6)

**Date**: 2026-05-20
**Scene**: `butterfly.spz` — 177,132 splats, 4,158 occupied voxels at `vox=64`
**Verdict**: **✅ partial** — the stack mechanism works mechanically (commit + undo + density cap + frame-rate target all met), but the camera-facing targeting strategy only finds an empty cell for ~60 % of source voxels on this scene, and the visual coherence question is unresolved (subjective, scene-dependent).

## What ran

`BenchRunner.runH3Stack` (Wave D.6 addition) was driven via `?bench=h3` against `butterfly.spz`. The bench:

1. Picks **200 deterministic source voxels** by sampling `centerHash.keys` (sorted) at stride `floor(4158 / 200) ≈ 20`.
2. For each source, synthesizes a "camera above the scene" (mesh-local `(0, 1e6, 0)`) so `resolveStackTargeting` picks +Y as the dominant face with X/Z fallbacks.
3. Runs a real `StackOp` against the production `BufferedPackedSplatsWriter` + `StackSlotPool` + `StackedSplatsHash`.
4. Counts a stack as committed if the op didn't bail on a density-cap-hit / capacity-exhaustion / all-three-axes-blocked failure.

[Raw data](data/2026-05-20-h3-vox64.json) — 4 KB.

Average per-op frame time across the full session: **8.33 ms** (≈ 120 fps potential, RAF-throttled).

Plan §5 Wave D required FPS > 30 (= < 33 ms p95) across the session. Recorded p95 stays under **10.6 ms** — well clear of the budget. Stack ops do not visibly cost more than carve ops; the writer's per-frame batching ([`packed-splats-writer.ts`](../../src/viewer/packed-splats-writer.ts), Wave D.4) keeps the GPU re-upload to one per frame regardless of how many `setSplat` writes landed.

## Targeting success rate

| Outcome | Count | % |
|---|---|---|
| Committed | **121 / 200** | 60.5 % |
| Refused (all 3 axis targets occupied, or density cap reached) | 79 / 200 | 39.5 % |
| Total splats stacked across 121 commits | **4,735** | ≈ 39 splats / commit |

The 39.5 % refusal rate is *not* a bug — it's a measured property of the camera-facing targeting strategy in a dense scene. For source voxels on the butterfly's body (interior of the silhouette), all three preferred axes (+X / +Y / +Z given the synthetic camera) often have an occupied neighbour, so there's no adjacent empty cell to fill. The interactive UX (key 3) doesn't show this number because the user only sees the *successful* hovers — the picker resolves to a surface voxel, the targeting picks the camera-facing empty cell, ghost appears. In a real hover session the user has fine control and almost always finds an empty face; the bench's deterministic sampling exposes the worst case.

## Visual coherence — the subjective question

The plan flagged "visual seam between source and target voxel" as the most likely H3-falsifier. Reading the captured screenshots and the live `?stackSmoke=1` (now removed) sessions during D.4 verification:

- **Cluster shape transfers cleanly**: each stacked cluster *is* the source voxel's splat set, translated. Source-to-target spatial relationships within the cluster are preserved exactly.
- **Per-splat colour / scale / rotation also transfer** via Spark's `forEachSplat` exposing all six params. The only thing not copied is the SH coefficient — the source splat's view-dependent shading is lost on the duplicate.
- **Seam**: at the boundary between the original splat set and the stacked cluster, there's a measurable density discontinuity (no smooth fall-off across the voxel boundary). On a uniformly-coloured surface (e.g., a stone wall) this would be barely visible; on a chromatic gradient (butterfly wing) it shows as a sharper-than-natural edge.

**Subjective rating (1–5, by author): 3.0 / 5** — "looks like additional wing material but you can tell it was placed rather than rendered." Not falsified, not visually seamless either.

Future work to improve visual coherence:
- Multi-source sampling — draw from a 3×3 region of source voxels rather than a single cell.
- Boundary opacity sigmoid — taper the stacked cluster's opacity over a half-cell at the source/target seam.
- SH transfer — extend Spark's `setSplat` (or use `getSplat` + raw-array write) to carry SH coefficients through the copy.

These are all Wave-D+ candidates; the MVP stack is good enough to call H3 "partial / works".

## The mechanical bits — verified

- **Undo/redo round-trip**: each `StackOp.undo()` zeros opacity on every allocated slot and releases them back to the pool; the next `do()` re-acquires (LIFO) and writes. The Wave D.1 test suite includes a "do → undo → do" assertion that the visible scene state (not the slot indices) is byte-equivalent.
- **Density cap = 200 splats per cell**: enforced in `handleStackPointerMove` before the provisional `StackOp.do()`; cursor turns gray (CURSOR_COLOR_FORBIDDEN), stats panel says "density cap (N/200)".
- **Capacity = 200,000 stack slots**: pre-allocated at load via `BufferedPackedSplatsWriter.preallocate(splatCount, 200_000)`. The bench session consumed 4,735 slots (~2.4 % of capacity).

## Limitations

- **39.5 % refusal rate** in the bench is real but not user-visible; UX-tested by the author, the failure rate is much lower under deliberate clicking (the user picks visible empty cells).
- **SH not copied** — the stacked cluster is slightly flatter in view-dependent shading than the source.
- **Camera-facing targeting only** — no "drill-in" or "behind-surface" stack modes.
- **Bench is single-scene** — `butterfly.spz` only.
