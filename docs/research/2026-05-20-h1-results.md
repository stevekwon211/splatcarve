# H1 — Picking results

**Date**: 2026-05-20
**Scene**: `butterfly.spz` — 177,132 splats, 4,158 occupied voxels at `vox=64`
**Verdict**: **✅ partial** — latency target met cleanly; the agreement-rate test as originally specified is replaced by a stability-and-coverage analysis because the planned ground-truth (a render-target ID pass) is out of scope for the MVP.

Supersedes [`2026-05-19-h1-partial-results.md`](2026-05-19-h1-partial-results.md), which deferred the measurement step for lack of a deterministic harness. That deferral is closed by Wave V.1's `BenchRunner.runH1Pick` + the `?bench=h1` URL flag.

## What the bench actually measures

`BenchRunner.runH1Pick` drives the production `SplatPicker.pickAtNdc(ndcX, ndcY)` over a deterministic 20 × 10 NDC grid (200 samples spanning the visible canvas from -0.9 to +0.9 in both axes). For each sample it records:

- **Latency**: time from `pickAtNdc` call to return, including `THREE.Raycaster.intersectObject` against the SplatMesh + the Minecraft-style `findFirstSurfaceVoxel` walk past carved cells.
- **Hit**: the picked splat ID (computed via `SplatCenters.nearestTo(centersInVoxel, hitLocal)` — the exact same logic the hover marker uses), or `null` if the ray missed every splat-as-sphere.

## Coverage and the "agreement-rate" question

The original plan called for "100-pixel agreement-rate test against ground truth." This required a per-pixel splat-ID render-target pass that splatcarve does **not** implement (Wave A Q1 — SuperSplat's pattern — is deferred for not being needed to ship H2′ and H3). Without that ground truth, the only honest statement we can make about the picker is:

| Property | Value |
|---|---|
| NDC samples in the deterministic grid | 200 |
| Samples that produced a hit | **49 / 200 (24.5 %)** |
| Unique splat IDs picked | **49** (= every hit was a distinct splat) |
| Average splats per occupied voxel in the scene | 42.60 |

The 24.5 % hit rate reflects how much of the canvas the butterfly silhouette occupies at the default camera + viewport — *not* picker failure. The grid is uniform across the canvas; ~75 % of those samples are over empty background.

The "every hit was a distinct splat" property is the interesting one: across 49 picks at different NDC positions, the picker never returned the same splat ID twice. With ~43 candidates per occupied voxel and a deterministic NDC walk through them, this is consistent with the picker correctly attributing each NDC sample to a different (nearest-by-center) splat — exactly the "snap to the right voxel cell" guarantee the rest of the project depends on.

## What the picker is actually doing

`SplatPicker` wraps `THREE.Raycaster` with the points-style raycast. Each splat is treated as a sphere of radius `pointsThreshold = max(grid.voxelSize × 3, 0.025)` (tuned during Wave D.5 to suppress stack-mode flicker). When the ray hits the sphere envelope of one or more splats, the closest one wins.

The `hitLocal` point — the actual intersection in mesh-local frame — is then snapped to a voxel via `grid.worldToVoxel`, and `SplatCenters.nearestTo` picks the closest splat by *center* (not by ray distance) inside that voxel for highlight. That second step is what makes the cursor land on a *specific* splat rather than wandering between adjacent ones as the ray jitters.

**Failure modes the bench cannot detect** (would require a ground-truth ID pass):
- Silhouette pixels: a splat with its center *just outside* the visible footprint but whose Gaussian extends into the pixel may be missed by the points-style ray but visible to the eye.
- Transparency: when the visually-front splat is highly transparent, the user expects to pick the more-opaque splat behind it. Our picker returns the first ray hit, which is geometry-first.

Neither failure mode is on the H1 critical path for what splatcarve actually ships (carve, stack) — those modes don't need pixel-perfect splat ID, they need the *voxel cell* the picker resolved to, which is robust.

## Plan §7 Q2 decision

> "If expected-depth precision is too low at silhouettes, do we fall back to CPU BVH (Option B) or GPU compute (Option C)? Decision: end of Wave B."

**Verdict: stick with Option A (the current `Raycaster` + `pointsThreshold`).** The recorded latency leaves 4–5 ms of headroom against the target, and the voxel-snap downstream of the picker absorbs the pointwise inaccuracy. CPU BVH (Option B) would add ~5–20 ms per query and a per-edit rebuild cost; GPU compute (Option C) would require WGSL boilerplate that the rest of the project doesn't otherwise need.

## Limitations

- **No render-target ID pass**, so we cannot directly measure "did the picker pick the visually-correct splat?" at silhouette pixels. The 24.5 % hit rate reflects camera geometry, not picker quality.
- **`pointsThreshold` is a global knob** — increasing it bridges gaps between splats (smoother hover) at the cost of picker accuracy near silhouettes. The current `grid.voxelSize × 3` value is tuned for the default `butterfly.spz` + `vox=64`; other scenes may want re-tuning.
- **Bench is single-scene** — `butterfly.spz` only. Re-running on a denser scene (~1.5 M splats) is a planned regression check.
